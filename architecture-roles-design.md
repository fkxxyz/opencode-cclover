# 架构角色系统设计文档

## 文档说明

本文档是对 `architect-role-requirements.md` 的初步设计,将原本单一的 Architect 角色拆分为 5 个专注的角色,并详细设计了它们的协作流程。

## 设计理念

### 为什么拆分?

原始的 Architect 角色职责过于复杂:
1. 与 TD 聊天讨论设计(对话思维)
2. 记录和整理数据(数据管理思维)
3. 分析趋势识别重构(统计分析思维)
4. 设计架构方案(架构设计思维)
5. 探索项目建立初始数据库(探索思维)

这些职责需要**不同的思维模式**,放在一个角色中会导致:
- 上下文过长,注意力分散
- 思维模式切换困难
- 效率低下

### 拆分方案

将 Architect 拆分为 **5 个专注的角色**:

1. **Architecture Consultant** - 架构顾问(已存在,需改动)
2. **Architecture Recorder** - 架构记录员(新增)
3. **Architecture Analyst** - 架构分析师(新增)
4. **Architecture Explorer** - 架构探索者(新增)
5. **Architecture Designer** - 架构设计师(新增)

## 核心概念

### 需求数据库

**路径**: `<project>/.cclover/architecture/requirements-database.md` (硬编码)

**用途**: 
- 记录所有任务的信息(需求、耗时、模块等)
- 存储统计分析结果(缓存)
- 追溯架构演化历史
- 支持重构决策

**记录粒度**: 每个派发下来的任务(Task Designer 设计的任务粒度)

### 重构决策模型

**核心原理**: 基于成本收益风险的量化分析

架构师是**绝对理性的决策者**,没有"痛苦"的感受,只有成本和风险的计算:

```
痛苦 → 成本/风险的翻译:
- 开发痛苦 → 变更成本增加 + 出错风险增加
- 维护痛苦 → 修复成本增加 + 回归风险增加
- 扩展痛苦 → 新功能开发成本增加 + 技术债务累积风险
- 理解痛苦 → 学习成本增加 + 误解风险增加
- 质量痛苦 → 测试成本增加 + 线上故障风险增加

决策模型:
当前不重构的总成本 = 短期继续开发成本 + 长期累积成本 + 各种风险的期望损失
重构的总成本 = 重构实施成本 + 重构期间的机会成本 + 重构风险的期望损失

如果: 当前不重构的总成本 > 重构的总成本
则: 提出重构建议
```

**数据来源**: 任务耗时数据(从批准到完成的绝对时间)

**量化分析示例**:
- "MessageService 最近 3 个需求平均耗时从 2 天增加到 5 天,增长 150%"
- "如果重构 MessageService 需要 5 天,但能让未来 10 个类似需求从平均 5 天降到 2 天,节省 30 天"
- "ROI = 30 天 / 5 天 = 6 倍"

### 增量思考机制

**问题**: 每次任务完成后重新分析整个数据库成本太高

**解决方案**: 缓存上次分析结果,只做增量更新

**机制**:
1. **读取上次分析结果**(缓存) - 各模块统计指标、趋势判断、潜在重构点
2. **增量更新**(只分析新任务) - 更新相关模块指标、重新计算趋势
3. **触发深度分析**(仅在必要时) - 某指标超过阈值、趋势持续恶化时才进行完整分析

**数据库需要存储的缓存**(具体格式待设计):
- 模块统计(平均耗时、bug 数量、变更频率等)
- 趋势判断(成本上升/稳定/下降)
- 最近 N 次任务耗时(用于趋势计算)

### 架构师成长阶段

**初期阶段**(数据积累期):
- 静默观察,不提重构建议
- 每个任务完成后更新数据库
- 建立基线数据

**中期阶段**(趋势识别期):
- 有了足够数据样本
- 开始识别成本趋势
- 但可能还不足以支撑重构决策

**成熟阶段**(主动决策期):
- 数据充足,趋势明显
- 能够量化分析成本收益
- 主动提出重构建议

## 角色详细设计

### 1. Architecture Consultant (架构顾问)

**职责**: 与 Task Designer 聊天,讨论任务设计的架构合理性

**触发时机**: Task Designer 发送需求文档 + 任务方案文档

**工作流程**:
1. **接收** - TD 发送需求文档 + 任务方案文档
2. **聊天** - 与 TD 对话式讨论设计问题
   - 可以访问需求数据库 (`<project>/.cclover/architecture/requirements-database.md`)
   - 可以访问代码、文档、任何资源
   - 使用 edit_tasks 跟踪多个问题(如果有多个)
3. **结束** - 聊天结束后:
   - 发送批准信号给 TD (简短,1-2 句话)
   - 雇佣 Architecture Recorder
   - 发送需求文档 + 任务文档给 Recorder 作为初始消息

**工具使用**:
- `send_message`: 与 TD 聊天,高频使用
- `edit_tasks`: 跟踪多个设计问题(照抄现有 AC 的用法)
- `hire_employee`: 聊天结束后雇佣 Recorder
- `create_agent`: 不使用

**canHire**: `["Architecture Recorder"]`

**相比现有 Architecture Consultant 的改动**:
1. ✅ 保留现有的聊天机制和 Silent Completion
2. ⚠️ **新增**: 聊天结束后雇佣 Architecture Recorder
3. ⚠️ **新增**: 明确可以访问需求数据库
4. ⚠️ **新增**: 数据库路径硬编码在系统提示词中

**系统提示词改动要点**:
- 在"Silent Completion"部分增加:聊天结束后,除了发批准信号给 TD,还要雇佣 Recorder
- 在角色描述中明确:可以访问需求数据库 `<project>/.cclover/architecture/requirements-database.md` 来了解历史信息
- 在 canHire 中增加 `Architecture Recorder`

### 2. Architecture Recorder (架构记录员)

**职责**: 记录任务信息到需求数据库,整理数据结构

**触发时机**: Architecture Consultant 雇佣并发送需求文档 + 任务文档

**工作流程**:
1. **接收** - AC 发送需求文档 + 任务文档
2. **读取数据库** - 读取现有需求数据库
3. **提取信息** - 从文档中提取:
   - 需求内容摘要
   - 涉及的系统模块
   - 任务开始时间(从 TD 或系统获取)
4. **记录条目** - 在数据库中添加新的需求记录
5. **整理结构** - 保持数据库格式一致,必要时重新组织
6. **通知 Analyst** - 记录完成后:
   - 雇佣 Architecture Analyst
   - 发送新增记录的信息给 Analyst

**工具使用**:
- `send_message`: 不使用(直接雇佣 Analyst)
- `edit_tasks`: 不使用(工作简单,不需要任务管理)
- `hire_employee`: 记录完成后雇佣 Analyst
- `create_agent`: 不使用

**canHire**: `["Architecture Analyst"]`

**requiredArgs**: 无

**关键设计点**:
- Recorder 只负责**记录**,不做分析
- 记录时**不知道任务完成时间**(任务还未开始执行)
- 任务完成时间由后续流程补充(待设计)
- 保持数据库格式一致性

### 3. Architecture Analyst (架构分析师)

**职责**: 增量分析统计数据,识别重构机会,判断趋势

**触发时机**: Architecture Recorder 雇佣并发送新增记录信息

**工作流程**:
1. **接收** - Recorder 发送新增记录信息
2. **读取数据库** - 读取需求数据库和缓存部分
3. **增量更新** - 只分析新任务:
   - 识别涉及的模块
   - 更新模块统计指标(平均耗时、bug 数量等)
   - 重新计算趋势(成本上升/稳定/下降)
4. **判断阈值** - 检查是否触发深度分析条件:
   - 某模块成本增长超过阈值(如 50%)
   - 趋势持续恶化(连续 N 次上升)
   - Bug 频率异常
5. **深度分析**(如果触发) - 完整的重构方案分析:
   - 计算当前不重构的总成本
   - 估算重构的总成本
   - 计算 ROI
6. **汇报建议**(如果识别到重构机会):
   - 向 Boss 发送重构建议
   - 包含数据支撑、成本收益分析、风险评估
7. **更新缓存** - 将分析结果写回数据库缓存部分

**工具使用**:
- `send_message`: 向 Boss 汇报重构建议
- `edit_tasks`: 不使用(分析工作不需要任务管理)
- `hire_employee`: 不使用(不雇佣其他角色)
- `create_agent`: 不使用

**canHire**: `[]`

**requiredArgs**:
```yaml
boss:
  type: string
  description: "Boss 的名字,用于汇报重构建议"
```

**关键设计点**:
- 使用增量思考机制,降低分析成本
- 只在必要时进行深度分析
- 分析结果缓存在数据库中
- 初期静默,数据不足时不提建议

**待设计细节**:
- 缓存数据结构
- 增量更新算法
- 触发深度分析的阈值
- 趋势计算方法
- 模块识别机制

### 4. Architecture Explorer (架构探索者)

**职责**: 项目初期探索,建立初始需求数据库

**触发时机**: 用户手动雇佣(项目首次引入架构角色系统时)

**工作流程**:
1. **自主探索**:
   - 阅读 `AGENTS.md`(开发指南)
   - 阅读 `README.md`(项目概述)
   - 阅读现有架构文档
   - 浏览代码结构
   - 查看 git 历史
2. **向用户提问**:
   - 项目核心目标和长期规划
   - 当前已知的架构问题
   - 历史架构决策及背景
   - 架构演进的期望和约束
3. **建立初始数据库**:
   - 创建 `<project>/.cclover/architecture/requirements-database.md`
   - 记录项目背景和目标
   - 记录现有架构概述
   - 记录已知的架构问题
   - 记录初步的架构演进方向
4. **完成汇报** - 向用户汇报初始数据库已建立

**工具使用**:
- `send_message`: 向用户提问和汇报
- `edit_tasks`: 可能用于跟踪探索任务
- `hire_employee`: 不使用
- `create_agent`: 不使用

**canHire**: `[]`

**requiredArgs**: 无

**关键设计点**:
- 只在项目初期使用一次
- 建立初始数据库后,后续由 Recorder 维护
- 探索深度由 AI 自主判断

### 5. Architecture Designer (架构设计师)

**职责**: 重构批准后,撰写架构设计方案

**触发时机**: Boss 批准重构建议后,手动雇佣

**工作流程**:
1. **接收** - Boss 发送重构批准消息和 Analyst 的分析报告
2. **读取数据库** - 读取需求数据库,理解当前架构问题
3. **设计方案** - 撰写详细的架构设计文档:
   - 问题分析
   - 新架构设计
   - 迁移方案
   - 风险控制
   - 实施计划
4. **创建文档** - 将设计方案写入文档(路径待定)
5. **启动重构**:
   - 雇佣 Task Designer
   - 发送重构需求文档 + 架构设计文档给 TD
6. **完成汇报** - 向 Boss 汇报设计完成,重构已启动

**工具使用**:
- `send_message`: 向 Boss 汇报
- `edit_tasks`: 可能用于跟踪设计任务
- `hire_employee`: 雇佣 Task Designer
- `create_agent`: 不使用

**canHire**: `["*Task*Designer*"]` (支持各种 Task Designer 变体)

**requiredArgs**:
```yaml
boss:
  type: string
  description: "Boss 的名字,用于汇报设计完成"
```

**关键设计点**:
- 专注于架构设计,不做分析
- 设计方案要详细可执行
- 雇佣 TD 启动重构流程

## 完整工作流程

### 正常任务流程

```
Task Designer 
  ↓ 发送需求文档 + 任务方案文档
Architecture Consultant (聊天讨论设计)
  ↓ 聊天结束
  ├─→ 发送批准信号给 TD
  └─→ 雇佣 Architecture Recorder
        ↓ 发送需求文档 + 任务文档
      Architecture Recorder (记录任务信息)
        ↓ 记录完成
        └─→ 雇佣 Architecture Analyst
              ↓ 发送新增记录信息
            Architecture Analyst (增量分析)
              ├─→ 更新统计指标
              ├─→ 判断是否需要深度分析
              └─→ (如果识别到重构机会) 向 Boss 汇报
```

### 重构流程

```
Architecture Analyst
  ↓ 识别到重构机会
  └─→ 向 Boss 汇报重构建议
        ↓ Boss 批准
      Boss 雇佣 Architecture Designer
        ↓ 发送批准消息 + 分析报告
      Architecture Designer (设计方案)
        ↓ 设计完成
        ├─→ 雇佣 Task Designer
        │     ↓ 发送重构需求文档 + 架构设计文档
        │   Task Designer (设计重构任务)
        │     ↓ ... 正常任务流程 ...
        └─→ 向 Boss 汇报设计完成
```

### 项目初期流程

```
Boss 雇佣 Architecture Explorer
  ↓ 
Architecture Explorer (探索项目)
  ├─→ 自主探索(读文档、代码、git 历史)
  ├─→ 向 Boss 提问
  ├─→ 建立初始需求数据库
  └─→ 向 Boss 汇报完成
```

## 需求数据库结构(初步)

```markdown
# 需求数据库

## 需求记录

### [需求ID] 需求标题
- **时间**: YYYY-MM-DD
- **类型**: 功能需求 / Bug修复 / 性能优化 / 重构
- **描述**: 简要描述
- **涉及模块**: 模块A, 模块B
- **任务开始时间**: YYYY-MM-DD HH:mm:ss
- **任务完成时间**: YYYY-MM-DD HH:mm:ss (初始为空,任务完成后补充)
- **实际耗时**: X小时/天 (根据开始和完成时间计算)
- **架构影响**: 对架构的影响评估
- **关联需求**: 相关的历史需求ID
- **决策记录**: 架构决策和理由

## 分析缓存 (由 Analyst 维护)

### 模块统计
- **模块A**:
  - 平均耗时: X天 (趋势: ↑/↓/→)
  - Bug数量: N个
  - 最近5次任务耗时: [t1, t2, t3, t4, t5]
  - 变更频率: 每月M次

### 上次分析
- 时间: YYYY-MM-DD HH:mm:ss
- 结论: 暂无重构需求 / 识别到潜在重构点

## 架构演化历史

### [日期] 架构决策
- **背景**: 触发决策的原因
- **决策内容**: 具体的架构调整
- **影响范围**: 受影响的模块
- **预期效果**: 期望达到的目标

## 重构建议

### [日期] 重构建议
- **问题描述**: 当前存在的问题
- **数据支撑**: 统计数据和趋势分析
- **建议方案**: 重构方案概述
- **成本收益**: 实施成本与预期收益(ROI)
- **风险评估**: 潜在风险
- **状态**: 待批准 / 已批准 / 已完成 / 已拒绝
```

## 待设计细节

### Architecture Analyst 的增量分析机制

需要详细设计:
1. **缓存数据结构** - 存储什么统计指标,如何组织
2. **增量更新算法** - 新任务完成后如何高效更新统计指标
3. **触发深度分析的阈值** - 什么条件下从增量更新转为完整分析
4. **趋势计算方法** - 如何判断成本上升/稳定/下降
5. **模块识别机制** - 如何从任务中识别涉及的模块

### 任务完成时间的补充机制

问题: Recorder 记录时任务还未开始,不知道完成时间

可能方案:
1. 任务完成后,TD 通知某个角色补充完成时间
2. Analyst 在分析时发现完成时间为空,主动查询并补充
3. 系统自动监听任务完成事件,自动补充

需要进一步讨论。

### 数据库格式规范

需要详细定义:
- Markdown 结构规范
- 各部分的格式要求

## 注意事项

1. **避免过度设计**: 先实现基本功能,再逐步优化
2. **数据驱动**: 所有决策基于真实数据,不凭感觉
3. **保持简单**: 每个角色职责单一,易于理解和维护
4. **文档优先**: 所有决策都要记录在数据库中
5. **持续迭代**: 根据实际使用情况不断调整和优化

## 总结

通过将单一的 Architect 角色拆分为 5 个专注的角色,我们实现了:

1. **职责清晰**: 每个角色只做一件事,做好一件事
2. **思维聚焦**: 每个角色使用单一的思维模式
3. **效率提升**: 减少上下文切换,提高工作效率
4. **易于维护**: 每个角色独立,易于理解和修改
5. **数据驱动**: 基于真实数据做决策,不凭感觉
6. **增量优化**: 使用缓存和增量更新,降低分析成本

这个设计为后续实现提供了清晰的蓝图。
